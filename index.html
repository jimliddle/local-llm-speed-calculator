Multi-user serving

With continuous batching (llama.cpp server --parallel, vLLM, SGLang, TabbyAPI) the weights are read once per step for the whole batch, but every user's KV cache is read and stored separately. Aggregate throughput climbs with batch size until compute or memory capacity becomes the wall. Assumes all users decoding at the context length set above; MLX/LM Studio are mostly single-stream today.

The model:

  • Memory time / step = (active_params × bits/8 + users × KV) ÷ effective_BW
  • Compute time / step = users × 2 × active_params ÷ (TFLOPS × utilization)
  • Step time = max(memory, compute) — the roofline. Each step yields one token per user.
  • KV is read in full every token per user and pinned to primary memory.
  • Weights fill primary, then spill to RAM, then SSD; each tier read at its own bandwidth.
  • Multi-GPU tensor parallel scales bandwidth and compute at ~85%; layer split adds only capacity for a single stream.

Ignores prefill, attention FLOPs (small next to weight matmuls until very long context), dequantization overhead, and scheduler/software overhead. An optimistic ceiling; treat as best-case. Apple/AMD TFLOPS defaults are rough estimates.
